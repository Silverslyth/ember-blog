01
Foundation

What Is Self-Hosting

Self-hosting means running software on hardware you control — a spare laptop, a Raspberry Pi, an old desktop — instead of relying on a cloud service someone else operates. The software does the same things: file storage, media streaming, password management, home automation. The difference is the machine running it is yours.

The appeal is concrete. You own your data. You pay once for hardware, not monthly for a subscription. Services don't disappear because a company gets acquired. And you learn infrastructure skills that transfer everywhere — Linux, networking, containers, security.

The Model
Local server → services → your devices

A machine on your network runs containerized services. Those services are accessible from your phone, laptop, or TV — either locally or through a secure tunnel you control. You are the only cloud.

Who this is for

Anyone comfortable with a terminal. You don't need to be a sysadmin. If you can follow instructions, run commands, and tolerate occasional troubleshooting, you can self-host. Most setups, once running, require almost no maintenance.

02
Infrastructure

Choose Your Hardware

The barrier to entry is lower than people assume. Anything that runs Linux can be a server. The right choice depends on what you want to run, how much power consumption matters, and what you already have available.

Entry Level
Raspberry Pi

~$35-80. Low power draw (~5W). ARM architecture. Great for lightweight services — DNS, dashboards, small apps. Struggles with transcoding.

Mid Range
Old Laptop / PC

$0 if you have one. x86_64 — full compatibility, no ARM surprises. Higher power draw. Best all-around starter hardware.

Serious
Mini PC / NAS

$150-400. Intel N100/N200 chips sip power but run everything. Dedicated NAS units add RAID. For when you're committed.

Start with what you have

The most common mistake is buying new hardware before knowing what you actually want to run. Use an old laptop or a spare machine first. If you're still running services three months later, upgrade with intention.

Storage Considerations

For the OS and containers, a small SSD is ideal — 60GB minimum, 120GB comfortable. For media or file storage, separate spinning HDDs are cost-effective. RAID is not a backup — it's redundancy. Plan both separately.

03
Operating System

Install the OS

For a home server, you want a stable, headless Linux installation — no desktop environment, just the OS and SSH access. The two most common choices for beginners are Ubuntu Server and Debian.

Ubuntu Server
Free

LTS releases every 2 years with 5-year support. Largest community, most tutorials written against it. Best default for beginners. Slight Canonical telemetry you can disable.

Debian Stable
FOSS

Slower package cycle but rock solid. No corporate backing, pure community. Slightly more manual setup. Good choice if you care about FOSS lineage or are already familiar with Debian-based systems.

Proxmox VE
Free

Hypervisor for running multiple VMs and LXC containers on one machine. Overkill for day one but powerful once you want isolation between services. Web UI for management.

Arch / EndeavourOS
FOSS

Rolling release — latest everything. Higher maintenance but AUR access is real. Good choice if your workstation is already Arch-based and you want a consistent environment.

Initial Server Setup

1
Flash and boot the ISO

Use Balena Etcher or dd to write the ISO to a USB drive. Boot from it and follow the installer — hostname, user account, SSH enabled, no desktop.

2
Set a static IP

Assign a fixed local IP via your router's DHCP reservation (preferred) or configure it in /etc/netplan/ directly. Your server's address should never change.

3
Harden SSH

Copy your public key to the server, then disable password authentication in /etc/ssh/sshd_config. Set PasswordAuthentication no. Restart SSH. Now you can only log in with your key.

4
Update and set up automatic security patches

Run apt update && apt upgrade, then install unattended-upgrades. Security patches will apply themselves. You don't want to babysit this.

bash — initial hardening
# Copy your SSH key from your local machine first ssh-copy-id user@SERVER_IP # On the server — disable password auth sudo sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' \ /etc/ssh/sshd_config sudo systemctl restart sshd # Auto security updates sudo apt install -y unattended-upgrades sudo dpkg-reconfigure --priority=low unattended-upgrades
04
Containers

Docker Basics

Docker is the standard way to run self-hosted services. Each service runs in its own isolated container — its own filesystem, its own dependencies, no conflicts with anything else on the system. You can add, remove, update, or break a service without touching anything around it.

Docker Compose is the layer on top: instead of running long docker run commands, you write a compose.yml file that describes all your services at once. One command brings everything up.

Install Docker

bash — Docker install (Debian/Ubuntu)
# Official install script — quickest for a home server curl -fsSL https://get.docker.com | sh # Add your user to the docker group (no sudo required) sudo usermod -aG docker $USER newgrp docker # Verify docker run hello-world

Your First Compose File

Create a directory for each service, put a compose.yml inside it, and bring it up with docker compose up -d. Here's the structure using Portainer — a web dashboard for managing all your containers.

yaml — compose.yml (Portainer)
services: portainer: image: portainer/portainer-ce:latest container_name: portainer restart: unless-stopped ports: - "9000:9000" volumes: - /var/run/docker.sock:/var/run/docker.sock - portainer_data:/data volumes: portainer_data:
bash — bring it up
mkdir -p ~/docker/portainer && cd ~/docker/portainer # paste the compose.yml here, then: docker compose up -d # Check it's running docker compose ps # Open in browser: http://SERVER_IP:9000
Essential Docker Commands

docker compose up -d — start services in background  ·  docker compose down — stop  ·  docker compose pull — update images  ·  docker compose logs -f — follow logs  ·  docker ps — list running containers

05
Services

Your First Services

Start with services that have obvious, immediate value. Running something you actually use daily is the fastest way to understand whether self-hosting is for you — and it builds confidence for more complex setups later.

Jellyfin
Free / OSS

Media server. Replaces Plex, no paywall, no account required. Works with Nuvio via the Jellyfin addon — your own streaming service for files you already have.

Vaultwarden
FOSS

Self-hosted Bitwarden-compatible password manager. Uses the official Bitwarden clients. Your passwords, your server. Immediate practical value.

Nextcloud
FOSS

File sync and collaboration. Replaces Google Drive / Dropbox. Syncs across devices with desktop and mobile clients. Also does calendars, contacts, and notes.

Pi-hole / AdGuard Home
Free

Network-wide DNS-level ad and tracker blocking. Set your router's DNS to point here. Blocks ads across every device on your network, including phones and smart TVs.

Comet
FOSS

Self-host your own Stremio/Nuvio torrent scraper. No rate limits, no shared infrastructure. Add your debrid key and point AIOStreams at your own instance.

Homepage
Docker

Dashboard for all your self-hosted services. One page with live status indicators, links, and service widgets. The first thing you open, the thing that tells you everything is up.

jellyfin  ·  running on :8096
vaultwarden  ·  running on :8080
adguardhome  ·  running on :3000 (UI) / :53 (DNS)
comet  ·  running on :8888
homepage  ·  running on :3001
06
Access

Networking & Remote Access

Services running on your home network are only accessible from inside it by default. If you want to reach Jellyfin or Vaultwarden from outside — from your phone on mobile data, from work — you need a way to expose them safely.

Option A: Tailscale (Recommended for Beginners)

Tailscale creates a private mesh VPN across all your devices. Install it on your server and your phone. They join the same private network. You access your services by their Tailscale IP from anywhere, without opening any ports on your router.

bash — Tailscale install
curl -fsSL https://tailscale.com/install.sh | sh sudo tailscale up # Your server now has a stable Tailscale IP (100.x.x.x) # Access services from anywhere: http://100.x.x.x:8096
Zero port forwarding needed

Tailscale punches through NAT without you touching your router config. For a home server, this is the cleanest, safest approach. The free plan covers 3 users and 100 devices — more than enough.

Option B: Reverse Proxy + Domain

If you want clean URLs like jellyfin.yourdomain.com instead of IP:port, run a reverse proxy. Caddy is the easiest — it handles HTTPS certificates automatically. You'll also need a domain pointed at your home IP, and either a dynamic DNS service or a static IP from your ISP.

caddyfile — reverse proxy config
jellyfin.yourdomain.com { reverse_proxy localhost:8096 } vault.yourdomain.com { reverse_proxy localhost:8080 } # Caddy handles TLS automatically — no certbot needed
Security note

If exposing services to the public internet, never expose SSH directly. Use Tailscale for SSH access, keep only 80/443 open on your router for web services, and always use strong passwords or disable password auth entirely on anything public-facing.

07
Going Further

What Comes Next

Once your first services are running and you've lived with them for a few weeks, you'll have a much clearer picture of what you actually want. These are the natural next moves, roughly in order of complexity.

✓
Backups — the most important thing you're probably skipping. Use the 3-2-1 rule: 3 copies, 2 different media, 1 offsite. Borgbackup or Restic for encrypted local + remote backups.
✓
Monitoring — Uptime Kuma gives you a dashboard showing every service's uptime. Alerts you when something goes down. Grafana + Prometheus for full metrics if you want to go deeper.
✓
Automatic updates — Watchtower pulls updated Docker images and restarts containers automatically. Set it to notify you rather than auto-update if you prefer control.
✓
Immich — self-hosted Google Photos replacement. Automatic phone backup, face recognition, search. One of the most actively developed self-hosted apps right now.
✓
Arr stack — Sonarr, Radarr, Prowlarr, and qBittorrent work together to automate finding, downloading, and organizing media. Pairs with Jellyfin to give you a fully automated media library.
✓
Proxmox — once you want to run multiple isolated environments, Proxmox as a hypervisor lets you carve your hardware into VMs and LXC containers cleanly.
The Mindset
Break things in staging, not in production.

Run experiments in a VM or a separate container stack before touching your main services. Keep your backups current. Document what you've changed. Self-hosting is an incremental skill — every break is a lesson you only pay once.
